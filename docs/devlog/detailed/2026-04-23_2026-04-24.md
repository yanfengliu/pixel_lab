- B1: picked option A (explicit per-commit write back into
  `editedFrames[frameIndex]`) over option B (alias paint buffer and
  editedFrames). Aliasing felt like the kind of subtle sharing bug
  that a future refactor would accidentally undo — the explicit write
  is trivial and obvious.
- I2/I7: picked the counter approach over "pass fresh RawImage shells
  down every prop chain". The counter lives in one place, bumps
  atomically with the state change, and is deterministic to test.
- I4: kept the preview buffer preallocated but stayed on the
  RawImage-rasterized path so Shift-to-fill previews stay visually
  identical to the committed pixels. The simpler `ctx.strokeRect`
  approach would lose brush-size fidelity during the drag.
- I6: commit-on-unmount over revert-via-before-snapshot. Users
  generally want to keep the pixels they've already laid down;
  undoing once is cheap if they disagree.
- I9: overshoot instead of null-return. Drawing primitives already
  clip, so letting the coordinate through is safer than a special
  path.
- UNDO_CAP = 200 matches the reviewer's recommendation; well below
  memory pain (tens of MB even on 512² strokes) and well above normal
  session depth.

**Notes:**
- No architecture drift. Dep graph still `app → ui → io → core`;
  `renderCounters` lives in UI store only (session-only, not
  serialized). No new npm deps.
- Lessons appended to `docs/learning/lessons.md`:
  `editedFrames`-as-projection vs. -as-snapshot + render-counter
  pattern for in-place pixel mutations.
- Plan deviations: none substantive. Consolidated doc's B1 "option A"
  and I4 "preallocate" were both the natural fit.

## 2026-04-24 — Scroll-wheel zoom + middle-button pan + ColorPanel scrollbar fix (team lead)

**Action:**
- User asked for scroll-wheel zoom and middle-button pan, and flagged
  an ugly horizontal scrollbar in the ColorPanel via screenshot. Three
  small changes in one branch.

**Implementation:**
- `src/ui/Shell.tsx`: viewport ref, two useEffects — one attaches a
  native non-passive `wheel` listener on `.canvas-viewport` so
  `preventDefault()` actually suppresses viewport scroll; the other
  binds window mousemove/mouseup for pan tracking. Zoom anchors on
  cursor: compute the content point under the mouse before zoom, then
  (in `requestAnimationFrame` after React commits the new zoom) adjust
  `scrollLeft`/`scrollTop` so the same content stays put. Middle-button
  handler stores start coords + initial scroll offsets; mousemove
  applies the delta inverted.
- `src/app/styles.css`: `.canvas-viewport.panning` sets `grabbing`
  cursor, `user-select:none`, `pointer-events:none` on descendants
  (so mid-drag scroll doesn't accidentally trigger paint).
- `.color-panel`: `overflow-x: hidden`, `min-width: 0`. Slot stacks get
  `flex: 1 1 0` + `min-width: 0` so they shrink instead of overflowing
  when the rail is narrow. Hex input goes `width: 100%` with
  `max-width: 56px`, `box-sizing: border-box`.

**Tests:**
- `test/ui/Shell.test.tsx` (new) — 7 tests:
  - wheel up/down adjusts zoom.
  - zoom clamps at min/max.
  - wheel preventDefaults (so viewport doesn't scroll too).
  - middle-button drag updates scroll position and sets `.panning` class.
  - left-button mousedown does NOT start a pan.
- `test/smoke/drawing-smoke.mjs` extended with three end-to-end checks:
  wheel zoom grows canvas width, middle-button drag changes viewport
  scroll, ColorPanel `scrollWidth <= clientWidth` (no horizontal
  overflow).

**Gate output:**
- `npx vitest run`: 274/274 pass (was 267; +7 Shell tests).
- `npx tsc --noEmit`: clean.
- `npx vite build`: succeeds (~144 kB gz).
- Smoke: real Chromium. After zoom from 4 → 5, canvas width went
  512 → 640 (128-px source). Middle-drag of (-70, -60) moved viewport
  scroll by (+70, +60). ColorPanel scrollWidth === clientWidth.

**Notes:**
- The native-wheel-listener pattern (non-passive + preventDefault)
  matters because React's synthetic `onWheel` is passive by default —
  preventDefault there is a no-op and the viewport would both zoom

- ColorPanel fix is defensive — the user's narrow rail made the two
  color slots overflow horizontally; making them shrinkable is the
  minimal change and matches Aseprite's own "let the panel collapse"
  behaviour at narrow widths.
